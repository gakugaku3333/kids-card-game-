# 「遊んで成長する」改善計画書 — ちびっこひろば（3歳）・ゲームひろば本家（6歳）

作成日: 2026-07-08
対象: 既存2サイトを開発哲学「**遊ぶことで成長する仕組みを構築する**」（`CLAUDE.md`）に照らして改善する
実行: 別セッションでこの計画書を読み込んで着手する。キラキラひろば（10歳版）計画とは独立に進められるが、5章の共通部品は10歳版 Phase 1a と設計を揃えること

---

## 1. 哲学に照らした現状監査

| 観点 | ちびっこひろば（3歳） | 本家（6歳） |
|------|--------------------|------------|
| 遊びが主、学びは結果 | ◎ 設計原則から合致（失敗なし・音声・ごほうび） | ▲ クイズ4種＋かたかなが**裸のドリル**（「たしざんクイズ」と名乗り、正解数を数えるだけ） |
| 成長が見える | ▲ シールは貯まるが「上手くなった」は見えない。親からも見えない | ▲ トークン残高以外に成長の可視化なし。自己ベスト・前回比・レベル感がない |
| 失敗は次の挑戦への入り口 | ◎ 失敗が存在しない設計 | ▲ 誤答復習が「ふくしゅう」という勉強語で、報酬面の魅力もない |
| 年齢ごとに最適化 | ◎ ToddlerShell で別設計済み | ○ 迷路等は良い。クイズ系が年齢相応の「物語・収集」を持たない |

結論: **本家のクイズ体験の変装と成長可視化が最大の改善余地**。ちびっこは方向性が既に正しいので「見えない成長を見える化する」「本人の成長に追従する」の2点に絞る。

## 2. 本家（6歳）の改善

### Phase A: 全ゲーム一括で効く低コスト改善（core/ 拡張が中心）

1. **自己ベスト・前回比をリザルトに標準装備**: `core/shell.js` の `showResult()` を拡張し、ゲームごとの自己ベスト（正解数/タイム/スコア）を `katakana_game_data` 内に記録・表示。「じこベストこうしん！🎉」演出を追加。shell採用ゲーム全部に一括で効く、費用対効果最大の一手。既存スキーマへのフィールド追加になるため後方互換を smoke-test（`tasks/smoke-test.md`）で確認する。
2. **れんぞく正解コンボ**: `games/quiz/engine.js` にコンボカウンタ＋演出（2連続で「すごい！」、5連続で画面キラキラ＋ボーナストークン）。テンポの良い「もう1問」を作る。
3. **「ふくしゅう」→「リベンジクエスト」化**: fb-review モードの名称・文言・演出を変更し、通常より高いトークンレートを設定。「まちがえた問題こそおいしい」という動機づけに反転させる。
4. **禁句の緩和版チェック**: 6歳には10歳版ほど厳密な禁句（2.5節）は課さないが、「テスト」「べんきょう」「ドリル」調の文言・説明文は排除し、ハブの desc も「ちょうせんしよう」調に統一する。

### Phase B: 収集と物語で「学びの変装」を進める

5. **かたかなカード図鑑**: かたかなクイズの正解でその文字の「カード」（絵札: ア=アイスクリーム等）が図鑑に貯まる。46音コンプが長期目標。QuizEngine の正解フックにコレクション保存を足し、図鑑ページを1枚追加する。
6. **算数クイズにキャラ進行**: 正解を重ねるとお供のキャラ（ショップで買えるアバターと連動）が成長・進化する演出。既存 `recordAnswer` の蓄積データを活用し、レベル閾値表示だけなら軽実装で済む。
7. **買ったものがゲームに登場する**: ショップで装備したアバターをクイズ画面・リザルトに表示して応援させる。トークンを「貯める→買う→終わり」から「買ったものと一緒に遊ぶ」に変え、経済を循環させる。
8. **ゲーム名の変装（本人の反応を見て）**: 「たしざんクイズ」→「たしざんモンスターたいじ」のようなリネームは Phase B 末に候補を作り、本人が今の名前に飽きているかを見て判断する。名前だけ変えて中身が裸のままだと逆効果なので、5〜7 の実装後に行う。

### Phase C: キラキラひろばからの逆輸入（10歳版 Phase 1a 完了後）

9. 10歳版で作る**テーマスキン基盤**（世界観画面・依頼文テンプレート・報酬演出）が安定したら、本家クイズにも軽量スキン（例: たしざん→おかしやさんのお手伝い）を適用する。二重投資を避けるため、本家 Phase C は10歳版の基盤完成を待つ。

## 3. ちびっこひろば（3歳）の改善

**3歳向け設計原則（`tasks/3歳向けゲームひろば計画書.md` 2節）は一切崩さない**。失敗なし・音声主体・タップ完結はそのまま。

### Phase A: 本人に見えない適応難易度

1. **裏側の自動レベル調整**: pick エンジン系（いろタッチ・かずタッチ・なかまさがし等）に、直近の正解率に応じて選択肢数（3→4）や数の範囲（1〜5→1〜10）を静かに広げる/戻す仕組みを入れる。本人には難易度の概念を一切見せない（レベル表示・選択画面を作らない）。「いつのまにか難しいものができている」を実現する、3歳版における成長の仕組みの本丸。調整状態は `chibikko_data` に保存。
2. **「はじめてできた！」の特別演出**: 新しい種類の問題（新しい動物・新しい数・新しい形）に初めて正解した時だけ、通常より豪華なキラキラ＋専用ほめ音声を出す。初回フラグを記録するだけで実装は軽い。

### Phase B: 成長の見える化（親向け）と成長への追従

3. **おうちのひとメニューに「せいちょうレポート」**: プレイ回数だけでなく「言える色が増えた」「数が1〜10まで数えられた」等を適応難易度の到達状態から自動生成して表示。親が成長を実感でき、声かけの材料になる。
4. **「成長したら解放」ゲームの準備**: 計画書の将来候補（ひらがなさがし・おつかいごっこ）を、適応難易度が一定水準に達したら**ハブに新しいゲームカードが現れる**形で解放する。「お姉さんになったから新しい遊びが来た」という体験にする。実装は解放条件の判定＋カード表示制御のみ先行し、ゲーム本体は次の拡張セッションで作る。

## 5. 実行順序と10歳版との関係

| 順 | 内容 | 備考 |
|----|------|------|
| 1 | 本家 Phase A（自己ベスト・コンボ・リベンジクエスト・文言） | 単独で着手可。`showResult` 拡張は10歳版でも使うので**先にやると10歳版が楽になる** |

| 4 | ちびっこ Phase B（せいちょうレポート・解放ゲーム） | |

各 Phase 終了時に commit & push → 実機確認。効果は Phase 4 と同様「本人が自発的に遊び続けるか」で測る。

## 6. 実行セッションへの申し送り

- まず `CLAUDE.md`（開発哲学）と本計画書、メモリ `project_architecture.md`・`project_lessons.md` を読むこと。
- 自己ベスト・図鑑・コンボの各共通部品は、10歳版計画書（`tasks/10歳向けゲームひろば計画書.md`）の共通スキン基盤と将来統合できるよう、ゲーム固有ロジックと分離して実装する。
- 文言変更・リネームは子供が今の名前に愛着を持っている可能性があるため、大きな変更（ゲーム名）はユーザーに確認してから行う。演出追加・報酬調整は確認不要で進めてよい。
- 適応難易度（ちびっこ）は「難しくなりすぎて泣く」が最悪の失敗。上げ幅は保守的に、下げ判定は敏感に設定する（例: 2連続不正解で即1段階戻す）。
