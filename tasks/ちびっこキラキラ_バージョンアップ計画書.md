## 2. 哲学に照らした監査 — 今回の改善軸

1. **遊びが主、学びは結果** — ちびっこ・キラキラは達成済み。**本家だけがクイズ4種を裸のドリル名・裸の出題のまま残しており、3サイト中で最も哲学から遠い**。キラキラで実証済みの変装手法（questSkin・依頼文テンプレート）を逆輸入して解消する。

3. **失敗は次の挑戦への入り口** — 本家・キラキラともリベンジクエストが「行けば得」であることをハブでもっと目立たせる余地。

### Phase C2: 本人に見える成長 — シールちょうの「ずかん」化

3歳に折れ線グラフは無意味。**集めたものに再会できる**形で成長を見せる。

1. **シールをタップすると名前を読み上げ**: シールちょうが眺めるだけの台紙から「言える名前が増えていく図鑑」になる。実装はシール定義に読み上げ語を足すだけ。
2. **「できたね」の記録シール**: はじめてできた演出（Phase A実装済み）の発火時に、通常シールと別枠の**金縁きろくシール**を貼る（「10まで かぞえられた」等を絵で表現）。本人が「わたしできたの」と親に見せられる媒体にする。せいちょうレポート（親向け）と同じデータ源から出すだけなので軽実装。
3. **プリンセスのおはなし解放**: 記録シールが貯まるごとに、ゆきひめ達の1枚絵＋短い音声のおはなしシーンが1つ解放される（紙芝居1画面レベル）。一番好きなものを一番のごほうびに置く原則の続き。イラストは Gemini 生成（`tasks/イラスト生成指示書.md` に章追加）、未着の間は絵文字プレースホルダ。

### Phase C3: 適応難易度の対象拡大

Phase Aで pick 系に入れた仕組みを、残りの認知系へ広げる（本人には一切見せない原則は不変）。

- ぺあさがし: 4〜6枚 → 調子に応じて8枚まで
- かたちはめ: 基本5形 → 慣れたら回転あり・類似形（△と▽）を混ぜる

- かずタッチ: 1〜10到達済みなら「あわせて いくつ？」（絵を数えるだけのたし算前段）を静かに混ぜる

### Phase 5: 経済の完結 — 「クーポンが使える」「クエストがおいしい」

1. **割引クーポンの実装**: ショッピングモールで集めたクーポンを `kirakira/shop.html` の購入時に消費して割引適用できるようにする（10歳計画書3章で予告済みの未実装機能）。「割合の学習 → 実利」という経済直結の要を完成させる。
2. **リベンジクエストの導線強化**: ハブに「リベンジクエスト◯件 — ボーナス中！」バッジを常時表示（対象0件なら非表示）。まちがいが「おいしい在庫」に見える動線にする。
3. **デイリーおしごと**: 1日1回「今日のおしごと3つ」（例: スイーツカフェで5問正解 / めいろEXで1面 / 何でも自己ベスト挑戦1回）をハブに提示、達成でボーナストークン。禁句原則（2.5節）遵守で「ミッション」「おしごと」と呼ぶ。**遊ぶ順番を強制しない原則**と両立させるため、報酬は控えめ・未達成のペナルティなし。

> **2026-07-13 実装完了**: Phase 5 の3項目すべて実装。
> 1. `kirakira/data/coupons.json` にクーポン定義（`discountPercent`付き）を一本化し、ショッピングモールと`shop.html`双方から参照。`Store.hasUnusedCoupon`/`useCoupon`で「図鑑には残るが割引適用は1回だけ」の記録/在庫分離設計にした（図鑑としての収集価値と実利のクーポン消費を両立）。
> 2. リベンジクエスト常時表示バナーは Phase 0 の時点で実装済みだったことを確認（`kirakira/index.html`の`#review-banner`+`checkReviewDue()`）。追加実装は不要だった。
> 3. `kirakira/data/daily-missions.json` + `core/dailyMissions.js`（badges.jsと同じmetric/threshold汎用判定パターン）でデイリーおしごと3つを実装。`core/store.js`に日付キーで自動リセットする`daily`フィールドを追加し、`shell.showResult()`から`Store.trackDailyPlay()`を呼んで全ゲーム共通でプレイ回数・正解数を計測する設計にした。
> 実機検証: クーポン適用で全アイテム価格が正しく割引され消費後は再表示されないこと、デイリーおしごとがハブ再訪時に自動達成・トークン付与・トースト表示されること、ショッピングモールのクーポン図鑑が消費後も表示され続けることを確認。次はPhase 6（成長が「線」で見える）。

### Phase 6: 成長が「線」で見える

## 8. 実行セッションへの申し送り

- まず `CLAUDE.md`（開発哲学）・本計画書・メモリ `project_architecture.md`・`project_lessons.md` を読むこと。
