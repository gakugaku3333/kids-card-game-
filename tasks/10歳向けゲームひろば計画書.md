# 10歳向けゲームひろば「キラキラひろば」計画書

作成日: 2026-07-08
対象: 10歳の女の子（小学4年生想定）
実行: 別セッションでこの計画書を読み込んで着手する
参考: `tasks/3歳向けゲームひろば計画書.md`（ちびっこひろば）と対になる「上方向」の展開

---

## 1. 目的とコンセプト

- 開発哲学は全サイト共通で「**遊ぶことで成長する仕組みを構築する**」（`CLAUDE.md` 参照）。本計画はその10歳向け実装。
- 本家「ゲームひろば」（かたかな・算数レベル）は10歳には簡単すぎ、見た目も幼い。**難易度と世界観を10歳の女の子向けに引き上げた別ひろば**を作る。
- コンセプトは「**自分のペースで上手くなっていくのが見える**」×「**勉強に見せない**」。3歳版が「失敗が存在しない」設計だったのに対し、10歳版は**挑戦・記録更新・収集**を楽しさの核にする。学習コンテンツは裸のクイズとして出さず、すべて世界観のある「クエスト」に変装させる（2.5節）。
- 対象領域: 算数（小4〜5: わり算筆算・分数・小数・角度・面積・割合・平均・体積）/ 漢字（小1〜5の読み）/ 理科（小4〜5）/ 社会（都道府県・県庁所在地・産業）/ 英語（単語タイピング）/ 図工（ドット絵）/ 音楽（リズム・作曲）/ 論理パズル・迷路 / 創造（コーデ・マイルーム）
- ちびっこひろば（`chibikko/`）と同様、**同一リポジトリ内のサブディレクトリ `kirakira/`** として構築する（GitHub Pages 上で `.../kirakira/` になる）。

## 2. 10歳女子向け設計原則（全ゲーム共通）

3歳版の設計原則をそのまま使ってはいけない。10歳版は逆方向に振る。

1. **文字で情報を出してよい**: 説明文・スコア・ランキング表示OK。音声読み上げは不要（効果音は残す）。
2. **失敗・制限時間・スコアを解禁**: タイムアタック、コンボ、ライフ制を積極的に使う。ただし失敗時も「今日のベストまであと少し」など**次の挑戦につながる見せ方**にする。罵倒的・煽り的な演出はしない。
3. **自己ベストの見える化**: 全ゲームでハイスコア/ベストタイムを localStorage に記録し、リザルト画面で「じこベスト更新！」を派手に祝う。他人との競争ではなく**過去の自分との競争**を軸にする。
4. **難易度は3段階選択制**（★かんたん/★★ふつう/★★★むずかしい）。★★★は大人でも歯ごたえがある程度まで上げる。迷路（本家版6x6〜14x14）の難易度カーブを基準にする。学習系クイズはさらに**学年切替（小4/小5）**を持ち、★とは独立に出題範囲を選べるようにする（学年×★の2軸。JSONのlevel構成で表現し、UIは既存QuizEngineのレベル選択を流用）。
5. **1プレイ2〜5分**: 3歳版より長くてよいが、宿題の合間に遊べる長さに収める。
6. **収集とカスタマイズを深くする**: トークン→ショップの経済は本家より品数・レアリティを増やし、「集めたくなる」「飾りたくなる」を作る（3章）。
7. **操作はタップ＋ドラッグ＋キーボード**: 10歳なら全部使える。タイピング系はPCキーボード前提のゲームがあってよい（ハブのカードに🖮アイコンで明示）。
8. **世界観は「ゆめかわ」**: パステル紫×ミント×白、星・リボン・ホログラム調のキラキラ。3歳版のような幼いプリンセス調ではなく、**文具や推し活グッズ的な「オトナかわいい」**に寄せる。フォントは丸ゴシック系（Zen Maru Gothic 等をローカル配置 or システムフォント）。絵文字プレースホルダ→Gemini（Antigravity）生成イラストの運用は3歳版と同じ（`tasks/イラスト生成指示書.md` に10歳版の章を追加する）。

### 2.5 「勉強に見せない」5原則（学習系ゲームの絶対ルール）

学習コンテンツが「ドリル」に見えた瞬間に自発性は失われる。学習系は全ゲームこの原則で包む。

1. **禁句**: 「クイズ」「ドリル」「テスト」「もんだい」「べんきょう」という言葉をゲーム名・UI・リザルトに一切使わない。すべて「クエスト」「おしごと」「ぼうけん」「ミッション」と呼ぶ。
2. **出題は世界観の中の「依頼」として出す**: 「48÷6は？」と裸で聞かない。「お客さんが6人。クッキー48まいをおなじ数ずつ分けると1人なんまい？」のように、カフェの注文・実験の手順・旅先のできごととして提示する（裏のQuizEngineカーネルは同じ。文章テンプレートに数値を差し込むだけ）。
3. **正解の報酬がゲーム内で意味を持つ**: 「正解→丸がつく」で終わらせない。正解するとレシピの材料が手に入る→スイーツが完成する→お店のメニューが増える、のように**正解が世界を進める**構造にする。トークンはその結果として貯まる。
4. **まちがいは罰ではなくヒント**: 不正解は減点せず「ヒントを見てもう一回」への分岐にする。ヒント後の正解も報酬を出す（少し減らすだけ）。誤答復習（fb-review）も「ふくしゅう」と呼ばず**「リベンジクエスト」**として出し、通常よりボーナス報酬を高くして「おいしいコンテンツ」に見せる。
5. **遊ぶ順番を強制しない・でも学習が一番おいしい**: どのゲームから遊ぶかは完全に本人の自由。そのうえで報酬レートを「学習系 > アクション系 > 創造系（創造系はトークン消費側）」に設計し、コーデ・マイルームを充実させたければ自然と学習系クエストに向かう経済にする。

## 3. ごほうびシステム: トークン＋コーデ＆マイルーム

本家のトークン/ショップを土台に、収集要素を深化させる。

- **通貨は本家と同じトークン制**だが、データは完全分離（localStorage キー `kirakira_data`）。
- ショップの品揃えを2系統にする:
  - **コーデアイテム**: マスコットキャラ（オリジナルの女の子 or 猫）に着せる服・アクセ・髪型。レアリティ3段階（ノーマル/レア/キラ✨）。キラはガチャ演出（トークン消費のランダム排出、ダブりは半額返金）で入手。
  - **マイルーム家具**: ハブ画面の背景がじぶんの部屋になっていて、買った家具・壁紙・ポスターを自由配置できる。「ひろばに来るたび自分の部屋が出迎える」体験を作る。
- **学習クエストと経済の直結**（2.5節の原則3・5の実装）: ショッピングモール（割合）の正解報酬は実ショップの**割引クーポン**、日本一周トラベルの報酬は**ご当地アイテム**（マイルーム装飾）、漢字ハンターは**カード図鑑**そのものが報酬。学習系だけが持つ限定報酬を作り、「コーデと部屋を充実させたいから学習クエストへ行く」動線にする。
- **アチーブメント（バッジ）**: 「スイーツカフェのメニューコンプ」「タイピング50語」など各ゲームに2〜3個。バッジ一覧ページで進捗が見える。コンプ欲を刺激する長期目標として機能させる。名称・文言もクエスト調で統一（2.5節の禁句ルール）。

## 5. 技術構成

- **ノービルド・ESモジュール・静的サイト**。`kirakira/` サブディレクトリ、既存リポジトリの main ブランチで GitHub Pages 公開（chibikko と同方式）。
- **core/ の流用方針**（実装セッション冒頭で要判断）:
  - `core/shell.js`（GameShell）・`core/sound.js`・`core/modal.js` は `../../core/` からの相対 import で**そのまま流用**したい。ただし GameShell と QuizEngine は内部で `core/store.js`（STORAGE_KEY 固定）に依存しているため、**store.js に名前空間（ストレージキー）を注入できる小さなリファクタ**を先に入れる。本家の挙動を変えないこと（デフォルトキーは現行のまま）を smoke-test で確認してから kirakira 側で `kirakira_data` を渡す。
  - リファクタが波及しすぎる場合は chibikko と同様に `kirakira/core/` へコピーしてフォークする（判断基準: 変更が store.js の初期化部だけで収まるか）。
- **QuizEngine**: `games/quiz/engine.js` を流用し、`kirakira/data/quizzes/*.json` を追加。分数・小数・割合・平均カーネルは既存5カーネルへの追加として本体側に実装（本家でも将来使えるため）。カーネルが8個を超えて肥大化しそうなら抽象化の切り方を見直す（教訓20）。理科・社会で使う**画像/SVG選択肢**が既存エンジンで表現できるかを Phase 1 冒頭で確認し、必要なら選択肢レンダリングだけ拡張する。
- **ディレクトリ構成**:
  ```
  kirakira/
  ├── index.html          # ハブ = マイルーム（家具配置された部屋の中にゲームカード）
  ├── badges.html         # アチーブメント一覧
  ├── core/store.js       # 名前空間注入版 or フォーク（上記判断による）
  ├── games/<name>/index.html
  ├── data/games.json / quizzes/*.json / picross/*.json / shop-items.json
  └── assets/             # ゆめかわ素材（Gemini生成PNG + コード側SVG）
  ```
- **テーマスキン層**: 学習系7本は「QuizEngine（出題・採点）＋テーマスキン（世界観画面・依頼文テンプレート・報酬演出・進行管理）」の2層構成にする。スキンは裸のクイズ画面より工数がかかるため、Phase 1 では**共通スキン基盤**（依頼文差し込み・正解→アイテム獲得演出・図鑑/マップ進行の保存）を先に1つ作り、各ゲームはその差分（背景・キャラ・報酬データJSON）だけにする。スキンの進行状態（お店レベル・訪問県・カード図鑑）は `kirakira_data` 内に持つ。
- **Firestore ログ**: 誤答復習（fb-review）は漢字・算数で有効に働くため**導入する**。ただしUI上は「ふくしゅう」ではなく**「リベンジクエスト」**（報酬ボーナス付き）として出す（2.5節）。ページに firebase compat の `<script>` 2本を忘れない（無いと無言no-op、教訓参照）。ユーザー識別は本家と別名にする。

## 6. 既存教訓の適用チェックリスト

実装セッションで必ず守ること（`memory/project_lessons.md` 由来）:

- [ ] preview 確認は eval/snapshot 主役・30秒超は2段階・キャッシュ問題はポート変更
- [ ] localStorage キーは `kirakira_data`（`katakana_game_data`/`chibikko_data` と衝突しない）
- [ ] グリッド移動のドラッグ操作は「指へ向かって壁沿いに1マスずつ」方式（迷路EX）
