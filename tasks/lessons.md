# 教訓 (Lessons)

子供用ゲームひろばの開発において得られた知見と教訓を記録します。

---

## 1. アセットと難易度の柔軟なマッピング
- **状況**: ユーザーから「バナナ、りんご、さくらんぼ、いちご」に加えて「ロップイヤー（垂れ耳うさぎ）」などのアセット追加の要望があった。
- **解決策**: 最大難易度（20枚=10ペア）に対してアセットが11種類となったため、ゲーム開始時に「全アセットから難易度に必要な数だけランダムに抽出してペアを作る」仕様に変更した。
- **教訓**: 固定されたアセット数とペア数を結びつけるのではなく、プールからランダム抽出する設計にすることで、アセット追加の柔軟性が増すだけでなく、プレイ毎に絵柄が変わるという「ゲームとしての再プレイ性の向上」につながる。

## 2. 外部依存のないWeb Audio APIによる効果音設計
- **状況**: ゲーム内でカードをめくる、揃う、外れる、クリアするなどのアクションに効果音が必要だったが、外部音声ファイルの読み込みはネットワーク環境やCORS、ファイルパスミスによるエラーリスクがある。
- **解決策**: Web Audio APIのオシレーター（三角波、正弦波、鋸歯状波など）とゲインノードを用いて、JavaScriptコード内で直接音響を合成した。
  - めくる: `triangle` 600Hz / 0.08s
  - 揃う: `sine` C5-E5-G5-C6 のアルペジオ
  - はずれ: `sawtooth`+`triangle` 180Hz / 175Hz の同時発音による不協和音
  - ファンファーレ: `triangle`+`sine` によるクリアメロディ
- **教訓**: シンプルな効果音であれば、外部ファイルを使わずにJS内で合成するアプローチが最も頑健であり、パフォーマンス的にも優れている。

## 3. モバイル表示でのスクロール防止設計
- **状況**: 神経衰弱ゲームは、画面内にすべてのカードがスクロールなしで収まることが極めて重要（スクロールが発生するとゲーム性が著しく損なわれるため）。
- **解決策**: CSS Gridと `aspect-ratio` を用い、ボードの最大高さを `100dvh - ヘッダー高` に設定して、`vmin` などの相対単位でカードが自動伸縮するようにした。また、スマートフォン（縦画面）とPC（横画面）でグリッドの行数・列数を動的に入れ替えるメディアクエリを設定した。
- **教訓**: ゲーム画面はスクロールを極力発生させないよう、境界を厳密に管理するビューポート設計が必要。

## 4. ユーザー意図の正確な理解: 「サムネイル画像」の指すもの
- **状況**: ユーザーから「ホーム画面のサムネイル画像を再作成して」というリクエストを受けた。
- **問題**: 「サムネイル画像」を「apple-touch-icon（Webクリップ用アイコン）」と解釈してしまい、ユーザーが実際に意図していた「ゲームカードのサムネイル画像（memory_game_thumb.png 等）」とのズレが生じた。自動承認されたため、確認なしに誤った方向へ進行し手戻りが発生した。
- **教訓**:
  - **ユーザー視点を第一に**: 「ホーム画面のサムネイル画像」と言われたら、まず画面上で最も目立つ画像要素（＝ゲームカードのサムネイル）を指していると解釈すべき。技術用語（apple-touch-icon等）への飛躍を避ける。
  - **自動承認下でも慎重に**: 実装計画が自動承認される環境では、Open Questionsを設けて止まるのではなく、最も自然な解釈で実行し、判断に迷う場合は直接テキストで質問する。

## 6. 横向き（ランドスケープ）でコンテンツが切れる問題

- **状況**: タイピングゲームで `body { overflow: hidden; align-items: center; }` を設定していたため、横向き時に縦の高さが足りず巨大フォント(240px)やキーボードが上下に切れて見えた。スクロールもできない状態。
- **解決策**:
  1. `overflow: hidden` → `overflow-x: hidden; overflow-y: auto; padding: 20px 0;` に変更してスクロールを許可。
  2. `@media (max-height: 700px)` を追加し、文字・キー・マージンを全て縮小（文字 240px→110px 等）。
  3. `align-items: center` → `align-items: flex-start` にして、収まらない場合も上端から表示＋スクロール可能に。
- **教訓**: 縦スクロールを完全に禁止するデザインは「縦が長いコンテンツ＋横向き」で必ず破綻する。ゲーム画面以外は `overflow-y: auto` を基本にし、横向き用の `max-height` メディアクエリで要素を縮小させる設計にする。

## 7. 複数ページにまたがるトークン経済の共通化

- **状況**: トークン経済（残高・ショップ・購入品）が `games/rakugaku/index.html` の3500行モノリスの中に閉じ込められており、タイピングゲームや神経衰弱はトークンを獲得できない「孤立」状態だった。
- **解決策**: `js/store.js`（`window.Store` グローバル）を作り、localStorage キーを据え置きで既存データを保全しながら全ページで共有する方式を採用。
  - `let gameData = Store.getData()` でライブ参照を共有 → 既存コードの `gameData.tokens += 2` などはそのまま動く
  - `saveGameData()` を `Store.save()` に置き換えるだけで永続化＋`tokens-changed` イベント発火による全ページ表示更新が得られる
  - `shopItems` 配列は `Store.SHOP_ITEMS` へ移設し重複を排除
- **教訓**: localStorageはオリジン単位（スキームとホスト）で共有される。パス（ディレクトリ）が異なっても同じキーに同じデータでアクセスできる。別ページへのモジュール共通化は `<script src="../../js/store.js">` の相対パス読み込みで十分機能する。ビルドツールや ES modules は不要。

## 8. Plan Mode での十分な探索と方針確認

- **状況**: 「融合させて」という抽象的なリクエストに対し、何を融合するのか・どこまでやるのかが不明確だった。
- **解決策**: Plan Mode で Explore エージェントを2本並列で走らせ、（1）メニュー二重構造の発見、（2）トークン経済がどのファイルに閉じ込められているかの把握、を完了してから AskUserQuestion で方針（融合範囲・トークン獲得対象・ショップUIの形）を確認した上で実装。
- **教訓**: 「融合」「最適化」「整理」などのメタ指示は実装前に必ず現状のアーキテクチャを把握し、具体的な方針（スコープ）をユーザーに確認してから着手する。Plan Mode の Explore エージェントを惜しみなく使って問題を構造化することが手戻り防止になる。

## 9. ゲームループのクリーンアップを startGame 冒頭で必ず行う

- **状況**: もぐらたたき・フルーツキャッチで、結果モーダルを閉じずに直接 `btn-start` を再度叩くと setInterval/requestAnimationFrame が2重に走り、スコアや落下速度が壊れた。
- **解決策**: `startGame()` の冒頭で `clearInterval(spawnTimer); clearInterval(countdownTimer)` / `cancelAnimationFrame(rafId)` を無条件で呼んで前回ぶんを掃除してから初期化する。
- **教訓**: タイマー・ループ系ゲームは「前回の状態が残っているかもしれない」前提で防御的に初期化する。UI で「スタート中はボタンを隠す」だけでは十分でなく、関数内部でも冪等に動くようにしておく。

## 10. SVGサムネイルはPNGより管理しやすい

- **状況**: 既存ゲームのサムネイルはAI生成PNG（バイナリ）だが、今回の3ゲームはSVGで新規作成した。
- **解決策**: `assets/images/{種別}_thumb.svg` でSVGを置き、`<img src>` から参照するだけで `object-fit:contain` の card-thumb として表示できる。
- **教訓**: 新規ゲームのサムネイルはSVG形式を第一候補にする。テキストエディタで直接編集でき、ファイルサイズが小さく、解像度非依存。既存PNGとの混在（`.png` / `.svg` 混在）は `<img src>` でそのまま共存できるので後から変えても問題ない。

## 11. preview_eval の30秒タイムアウトとゲーム時間の衝突

- **状況**: 30秒ゲームを `new Promise` で完走待ちしようとしたら preview_eval 自体の30秒制限に引っかかりタイムアウト。
- **解決策**: ゲームを非同期起動してから eval を返し（タイムアウトを避ける）、完了後に別の eval 呼び出しで `result-modal.classList.contains('show')` や localStorage を読んで結果を確認する2段階方式にする。
- **教訓**: preview_eval は 30 秒で強制終了する。それ以上かかる非同期処理は「起動だけして返す → 後で状態確認」に分割すること。

## 12. preview_start で同じポートを再利用するとブラウザが古いJSをキャッシュし続ける

- **状況**: `store.js` に新API（`equip`, `getAvatar`, `slotOf`）を追加したが、`preview_start` で同じポート(8765)を止めて再起動しても `typeof Store.equip` が `"undefined"` のまま変わらなかった。`avatar.js`（新規ファイル）は問題なく読まれるのに `store.js` だけ古い版が使われ続けた。
- **原因**: 既存ファイルへの変更はブラウザが aggressive にキャッシュする。新規ファイルはキャッシュに無いため必ず取得される。
- **解決策**:
  1. `fetch('js/store.js?cb='+Date.now()).then(r=>r.text()).then(t=>t.indexOf('function equip')>-1 ? 'DISK_OK' : 'DISK_OLD')` でサーバ上のファイルが更新済みかを確認（ディスクは正しかった）。
  2. `.claude/launch.json` のポートを `8765→8766` に変更して `preview_start("static")` を再実行 → ブラウザが初回取得するため全ファイルが最新に。
  3. 確認後ポートを 8765 に戻す。
- **教訓**: 「修正を加えた既存ファイルがブラウザに反映されない」場合はポートを変えて新鮮な origin を作ることで確実にキャッシュを回避できる。`location.reload()` では不十分なことがある。`eval(fetchedText)` でインラインに注入して動作確認する方法（スクリプト再評価）も有効だが、本番テストはポート切替が確実。

## 13. preview_screenshot は固定グラデーション背景のページでblankになることがある

- **状況**: 着せ替えモーダルを開いた状態でスクリーンショットを撮ったが、何度撮っても真っ白（薄ピンク一色）の画像しか取得できなかった。
- **原因**: `preview_screenshot` は CSS `background-image: linear-gradient(...)` や `animation` が絡む固定背景レイアウトでキャプチャタイミングがズレることがある。DOM自体は正しく描画されている。
- **解決策**: 表示確認は `preview_snapshot`（アクセシビリティツリー）と `preview_eval` で DOM の innerHTML・classList・getBoundingClientRect を直接読む。これで構造・テキスト・レイアウトの全てが確認できる。スクリーンショットは補足的なもので、動作確認の主軸にしない。
- **教訓**: 動作確認の主役は `preview_eval` + `preview_snapshot`。`preview_screenshot` は補足証拠として使い、blank でも慌てず DOM で確認する。

## 5. サムネイル画像管理: 1ゲーム1画像の原則
- **状況**: 当初、算数クイズ4種（たしざん・ひきざん・かけざん・わりざん）が同一の `math_thumb.png` を共有し、タイピング2種（こもじ・おおもじ）が同一の `rakugaku_thumb.png` を共有していた。
- **問題**: ゲームの区別がつきにくく、ユーザーから全てのサムネイルを個別にしたいとの要望があった。
- **解決策**: 全9枚のゲームカードに対し、それぞれゲーム内容を反映した固有のサムネイル画像をAI生成し、ファイル名も内容を表す命名規則（`addition_thumb.png`, `subtraction_thumb.png` 等）に統一した。
- **教訓**: 新しいゲームを追加する際は、**必ず固有のサムネイル画像を用意する**。共用は視認性を損なう。命名規則は `{ゲーム種別}_thumb.png` に従う。

## 14. 絵文字ベースのレイアウトへの画像アセットのシームレスな統合
- **状況**: アバターやショップのアイコンとして絵文字（テキスト）が使われていたシステムに、新しく画像アバター（ファイルパス）を追加する必要があった。
- **解決策**: 画像パスかどうかを自動判定する `Store.getIconHtml` ヘルパーを新設して `<img>` タグを出力するようにし、CSSで画像アセットに `width: 1em; height: 1em; object-fit: cover; border-radius: 50%;` を指定した。
- **教訓**: 絵文字ベースのレイアウト（`font-size` で要素の大きさを制御している箇所）に画像アセットを組み込む場合、 `width: 1em; height: 1em;` を設定することで親要素の `font-size` と自動的に同期させることができる。これにより、ヘッダーの小さなチップ（20px）や着せ替え画面の大きなプロフィール（92px）など、複数サイズでの表示崩れを防ぎつつ、最小限のコード修正で絵文字と画像を統合可能になる。

## 15. モーダルの開閉クラスはコンポーネントごとに違う（`.active` と `.show`）
- **状況**: 学習履歴モーダルを `.store-modal` として新規追加した際、開閉に `.active` クラスを使ったが `display:none` のまま開かなかった。`classList.contains('active')` は `true` を返すのに `getComputedStyle().display` は `none`、という症状で気づいた。
- **原因**: このプロジェクトは2系統のモーダルが混在している。
  - `.g-modal`（ログイン・準備中・ヘルプ）→ 表示クラスは **`.active`**（`gallery.js` 管理、CSSは `gallery.css`）
  - `.store-modal`（ショップ・きせかえ・きろく）→ 表示クラスは **`.show`**（`store.css` の `.store-modal.show { display:flex }`）
- **解決策**: `.store-modal` には `.show` を付け外しする。新規モーダルを足すときは、流用元のクラス（`.g-modal` か `.store-modal` か）に対応する表示クラスを必ず確認する。
- **教訓**: 「クラスは付いているのに表示されない」ときは、まず `getComputedStyle(el).display` を確認し、そのコンポーネントの表示用クラス名（`.active` / `.show` / `.open` など）がCSS側の定義と一致しているかを照合する。同種に見えるUIでも管理クラスが分かれていることがある。

## 16. 記録機能を作る前に「すでに記録されていないか」を確認する
- **状況**: 「学習の履歴が残るようにして」という依頼に対し、まず実装を始める前に既存コードを調査したところ、`firebase-log.js` の `logSession()` が**すでに**全クイズのセッション（正解数・問題数・トークン・日時）をFirestoreに保存していた。ただし**それを読み出して表示する画面が一切なかった**ため、データは溜まる一方で誰の目にも触れていなかった。
- **解決策**: 新規に保存ロジックを作るのではなく、`getSessions(limit, cb)` という**取得API**と表示モーダルだけを追加した。保存側は既存のまま活用。
- **教訓**: 「履歴・ログ・記録を残して」系の依頼は、**書き込み（保存）と読み出し（表示）のどちらが欠けているのか**を最初に切り分ける。既に保存されているケースは多く、その場合に保存ロジックを二重実装すると無駄＋データ不整合の元になる。`grep` で `logSession`/`add(`/`collection(` 等を先に探すこと。なお `logSession` を呼ぶのは現状クイズ2種（katakana・math-*）のみで、他ゲームは未対応（履歴に出ない）＝拡張余地として認識しておく。

## 17. 「移行/削除してよいか」はコードの存在ではなく到達可能性で判断する
- **状況**: `games/rakugaku/index.html`（3570行）には、かたかな/算数クイズの他に15個のミニゲーム・独自のショップ/とうけい/ゲーム一覧画面が同居していた。パッと見は「大きな機能一式」に見えるが、実際に削っていいか判断が必要だった。
- **調査方法**: `grep -rl "rakugaku"` でリポジトリ全体を検索し、`data/games.json` と `index.html` が実際にリンクしているURL（`?mode=katakana` 等、クエリパラメータ付き）を確認。rakugaku内部のURL無しトップ画面（`startScreen`）に到達する経路がハブのどこにも存在しないことを突き止めた。URL付きで直接アクセスするとJSの起動時ディスパッチが`startScreen`を素通りしてすぐ別画面を表示するため、`startScreen`上のボタン（🎮ゲーム/🛍️ショップ/📊とうけい）はユーザーが通常操作で辿り着けない＝事実上の死んだコードだった。
- **教訓**: 巨大ファイルの一部を「削ってよいか」「移行対象に含めるべきか」を判断する際は、**コードが存在すること＝到達可能であること、ではない**。エントリーポイント（今回は `data/games.json` とハブの `index.html`）からの実際のリンク・クエリパラメータを`grep`で洗い出し、どの画面が現実にユーザーへ到達するかを先に確定させてから削除・移行の範囲を決める。憶測で「念のため残す」を選ぶと、死んだコードを延命させたままリファクタの効果を薄めてしまう。

## 18. core/shell.js への移行はゲームごとに判断する（一律強制しない）
- **状況**: memory/catch/draw の3ゲームを `core/shell.js` に移行する際、全ゲームに同じ移行度合いを機械的に適用しようとすると問題が起きた。
  - memory: 独自の効果音・詳細な結果モーダル（タイム/ミス数）を持っており、shellの汎用リザルトモーダルに寄せると情報量が落ちる。
  - catch: `requestAnimationFrame` ループの冪等ガードを自前で正しく実装済みだったが、「一貫性のため」と shell の `requestAnimationFrame()` ラッパーに置き換えたところ、ループ内部の再帰呼び出し（`rafId = requestAnimationFrame(loop)`）は生のAPIのままだったため、shell側の追跡IDと実際に動いているフレームIDがズレて二重ループのリグレッションを埋め込みかけた（コミット前に検知して自前ガードに戻した）。
- **教訓**: `core/shell.js` は「便利な部品セット」であり「全ゲーム共通の型」ではない。(1) 独自実装の方が情報量・UXで優れている部分（結果モーダル等）はそのまま残し、shellからは`store`/`fireLog`/`sound`だけ使う、(2) 既にバグなく動いている冪等ガードがある場合はshellのタイマー追跡ヘルパーで無理に置き換えない、という判断をゲームごとに行う。「統一のための統一」はリグレッションの元になる。

## 19. トークン加算は「セッション完了時に一括」で統一する（shell経由ゲームの規約として）
- **状況**: 旧rakugakuの実装は正解するたびに即座に `Store.addTokens(2)` していたが、`core/shell.js` を使う新しいゲーム群（memory/catch/draw、および今回のgames/quiz/）は正解数を貯めておいて `shell.showResult({tokens})` を呼ぶ1回だけトークンを加算する設計になっている。
- **理由**: `shell.showResult()` は内部で `Store.addTokens(tokens)` と `FireLog.logSession(...)` を両方呼ぶため、これとは別に途中で `addTokens` を呼ぶと二重加算になる。
- **教訓**: `core/shell.js` を使うゲームでは、**トークンは即時加算せず貯めておき、最後に `shell.showResult()` へまとめて渡す**のが規約。プレイ中のリアルタイム加算演出が欲しい場合でも、`shell.store.getTokens()` を読んで表示だけ更新し、実際の `addTokens` 呼び出しは `showResult` に一本化する。

## 20. データ駆動化は「範囲を絞った汎用インターフェース」を作ると効く
- **状況**: `games/rakugaku/index.html` の算数出題ロジック（たしざん/ひきざん/かけざん/わりざん × 各3〜4レベル）は、type/levelごとのif/elseの羅列だった。これを闇雲に「全部JSONにする」と、JSON側にJavaScriptの計算式をそのまま書き写すだけの無意味な抽象化になりがちだった。
- **解決策**: 実際の生成パターンを洗い出すと `sum`（両方独立range）・`diff`（引く数が引かれる数未満）・`product`・`quotientExact`（割り切れる）・`quotientRemainder`（あまりを問う）の5種類の「生成カーネル」に収束した。JSON側は各レベルに `kind` とrange配列（`num1Range`/`num2Range`/`divisorRange`/`quotientRange`/`choiceRange`）を書くだけで済み、エンジン側は5関数の小さなディスパッチテーブルのみで完結した。
- **教訓**: 「データ駆動化」は生成ロジックを丸ごとJSONに押し込むことではなく、**繰り返し現れる生成パターンを少数の汎用カーネルとして抽出し、JSON側はそのカーネルへのパラメータだけを持つ**設計にすると、新しい学習コンテンツをJSON1個で追加できる本来の目的（保守性・再利用性）を達成できる。カーネルの数が肥大化するようであれば、それは抽象化の切り方が悪いサイン。

## 21. `core/shell.js` のリザルトモーダルは `css/store.css` に表示制御を依存していた（潜在バグ）
- **状況**: `games/maze/index.html`（本家版・GameShell使用）を新規作成し、3ラウンドクリアで `shell.showResult()` を呼んだが、`#shell-result-modal` に `.show` クラスが付与されているのに画面上は非表示のままだった（`getComputedStyle().display` が `none`）。
- **原因**: `#shell-result-modal` は `class="store-modal"` を持つが、`.store-modal { display:none } .store-modal.show { display:flex }` という表示制御ルール自体は `css/store.css` にしか定義されていなかった。`css/store.css` はハブ（`index.html`）だけが `<link>` しており、個別ゲームページ（`games/*/index.html`）は読み込んでいない。既存ゲーム（catch/clock等）がこれまで問題なく動いていたのは偶然ではなく、[[project-lessons]] 15番の「モーダル表示クラスがコンポーネントごとに違う」問題と表裏一体で、たまたま個別ページ側のCSSと衝突せず「見えないバグ」として温存されていた可能性がある。
- **解決策**: `core/shell.js` の `injectStyles()` 内に `#${RESULT_MODAL_ID}` の `display:none/flex` ルールを追加し、`css/store.css` の有無に依存せず shell 単体で完結するようにした。
- **教訓**: 「共通コンポーネント（shell.js等）が生成するDOM」の見た目が、実は**呼び出し側ページが読み込む外部CSSに依存している**ケースがある。新規ページ作成時に「前例のページと同じ import 一式」を機械的にコピーせず、`core/*.js` が要求するCSS依存を洗い出し、shell自身にカプセル化されているか確認する。今回のように「たまたま動いていた」ケースは新規ページ追加のタイミングで初めて露呈するので、新規ゲーム追加は既存の潜在バグを検出する良い機会でもある。

## 22. 「3歳向け」と「本家（年齢制限なし）」で同じテーマのゲームを作る場合は難易度設計を意図的に分ける
- **状況**: 「迷路ゲームを追加して」→「3歳用じゃない方にも迷路を追加して。レベルも上げたものを作成して」という2段階の依頼だった。
- **解決策**: ちびっこひろば版（`chibikko/games/maze/`）は3ラウンド固定・4x4〜6x6の小型迷路・ToddlerShell（文字なし、音声のみ、常に成功扱い）。本家版（`games/maze/`）はレベル選択制（6x6〜14x14の3段階×各3ラウンド）・GameShell（トークン経済・タイマー表示）とし、コード自体は似た迷路生成ロジック（穴掘り法によるBFS到達可能性保証）を流用しつつ、UIとゲーム設計は別々に最適化した。
- **教訓**: 同一ジャンルのゲームを2つの対象年齢向けに作る際は、**アルゴリズム（迷路生成・当たり判定等）は共有できても、難易度カーブ・UI・進行フロー（レベル選択の有無等）は年齢層ごとに独立して設計する**。片方を単にリサイズしただけでは「3歳向けには難しすぎる」「本家には物足りない」のどちらかに倒れる。

## 23. canvasベースのタッチ操作は「指に追従」ではなく「指へ向かって壁沿いに近づく」方式が安全
- **状況**: 迷路ゲームでキャラクターを指でドラッグ操作する際、素朴に「指の位置に即座にワープさせる」と壁を無視してすり抜けてしまう。
- **解決策**: `pointermove` のたびに、現在セルから隣接4方向のうち壁がなく（`canMove`）かつ指の位置に最も近づく方向へ**1マスずつ**移動する処理を、1イベントあたり数ステップ（今回は3〜4回）繰り返すロジックにした。これにより速いスワイプでも複数マスを壁沿いに正しく通過できる。
- **教訓**: グリッド制約のある移動系ゲーム（迷路・パズル）でポインタ操作を実装する場合は、「目標位置に瞬間移動」ではなく「壁を考慮した最短距離のセルへ段階的に近づける」ループにすると、操作感を保ちながら壁抜けバグを防げる。
